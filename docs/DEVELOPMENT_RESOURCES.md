# 开发资源管理规范

## 📂 目录用途

### `scripts/` - 开发辅助脚本

**用途**: 存放所有不参与正式编译的测试、验证、迁移等辅助脚本。

**适用文件类型**:
- ✅ 功能测试脚本 (`test_*.go`)
- ✅ 验证工具 (`verify_*.go`)
- ✅ 数据迁移脚本 (`migrate_*.go`)
- ✅ 单元测试文件 (`*_test.go`)
- ✅ 数据导入导出工具
- ✅ 批量处理脚本

**技术要求**:
```go
// 所有脚本必须在开头添加构建标签
//go:build ignore

package main
```

**命名规范**:
- 文件名使用小写 + 下划线分隔
- 测试脚本：`test_<功能>.go` (如：`test_ip.go`)
- 验证脚本：`verify_<对象>.go` (如：`verify_db.go`)
- 迁移脚本：`migrate_<描述>.go`
- 工具脚本：`tool_<用途>.go`

---

### `docs/` - 技术文档

**用途**: 存放项目开发过程中的技术文档、使用指南、最佳实践等。

**适用文件类型**:
- ✅ 快速入门指南 (`QUICKSTART.md`)
- ✅ 技术集成文档 (`SQLITE_INTEGRATION.md`)
- ✅ 配置说明 (`CONFIG_GUIDE.md`)
- ✅ 部署手册 (`DEPLOYMENT.md`)
- ✅ API 文档 (`API_REFERENCE.md`)
- ✅ 问题排查指南 (`TROUBLESHOOTING.md`)
- ✅ 架构设计文档 (`ARCHITECTURE.md`)

**不适用文件**:
- ❌ 临时修复记录（应整合到 Issue 或 PR）
- ❌ 个人笔记（应放在项目 Wiki）
- ❌ 过时的废弃文档（应及时删除）
- ❌ 重复的功能说明

**命名规范**:
- 文件名使用大写 + 下划线分隔
- 格式：`<主题>_<类型>.md`
- 示例：`SQLITE_GUIDE.md`, `API_REFERENCE.md`

---

## 🔄 工作流程

### 创建新脚本

1. **创建文件**: 在 `scripts/` 目录下创建 `.go` 文件
2. **添加标签**: 在文件开头添加 `//go:build ignore`
3. **实现功能**: 编写脚本逻辑，确保有清晰的注释
4. **更新文档**: 在 `scripts/README.md` 中添加说明
5. **测试验证**: 运行 `go run scripts/<脚本名>.go` 测试

**示例**:
```go
//go:build ignore

package main

import "fmt"

// CheckHealth 检查系统健康状态
func main() {
    fmt.Println("检查系统健康...")
    // 实现检查逻辑
}
```

---

### 创建新文档

1. **评估必要性**: 确认文档对开发者有价值
2. **创建文件**: 在 `docs/` 目录下创建 `.md` 文件
3. **遵循模板**: 使用统一的文档结构
4. **更新索引**: 在 `docs/README.md` 中注册
5. **审查内容**: 确保无拼写错误和过时信息

**文档模板**:
```markdown
# <文档标题>

## 概述
简要说明文档目的和适用范围

## 前置条件
列出阅读前需要掌握的知识

## 主要内容
分章节详细说明

## 示例代码
提供可运行的代码示例

## 常见问题
FAQ 解答

## 相关文档
链接到其他相关文档
```

## 🧹 维护规范

### 定期清理

**每月检查**:
- [ ] 删除不再使用的脚本
- [ ] 更新过时的文档
- [ ] 合并重复的内容
- [ ] 验证所有脚本仍可正常运行

**发布前检查**:
- [ ] `docs/` 只保留核心文档
- [ ] 移除所有临时性、过程性文档
- [ ] 确保 `README.md` 是最新的
- [ ] 验证所有示例代码可运行

---

### 版本控制

**Git 提交信息规范**:
```bash
# 添加脚本
git add scripts/verify_ssl.go
git commit -m "feat(scripts): 添加 SSL 证书验证脚本"

# 添加文档
git add docs/SSL_SETUP.md
git commit -m "docs: 添加 SSL 配置指南"

# 更新文档
git commit -m "docs(sqlite): 更新 SQLite 性能优化建议"
```

## 📊 质量标准

### 脚本质量

- ✅ **可执行性**: 所有脚本必须能独立运行
- ✅ **错误处理**: 完善的错误提示和恢复机制
- ✅ **注释完整**: 关键步骤有清晰注释
- ✅ **参数验证**: 输入参数有合法性检查
- ✅ **输出友好**: 控制台输出易于理解

### 文档质量

- ✅ **准确性**: 内容与实际功能一致
- ✅ **完整性**: 覆盖所有重要方面
- ✅ **可读性**: 结构清晰，语言简洁
- ✅ **实用性**: 提供实际可用的示例
- ✅ **时效性**: 及时更新反映最新状态

---

## 🎯 最佳实践

### 脚本开发

1. **单一职责**: 每个脚本只做一件事
2. **配置外部化**: 路径、参数通过环境变量或参数传递
3. **幂等性**: 多次运行结果一致
4. **无副作用**: 不修改生产数据，除非明确要求
5. **自包含**: 最小化外部依赖

### 文档编写

1. **读者导向**: 从使用者角度组织内容
2. **循序渐进**: 从简单到复杂
3. **图文并茂**: 适当使用图表、截图
4. **示例驱动**: 用实际例子说明概念
5. **交叉引用**: 链接到相关资源

## 🚨 常见错误

### ❌ 错误做法

1. 将测试脚本放在根目录
2. 文档缺少必要的构建标签
3. 在 `docs/` 存放个人笔记
4. 脚本没有错误处理
5. 文档长期不更新

### ✅ 正确做法

1. 所有脚本统一在 `scripts/`
2. 必须有 `//go:build ignore`
3. 只保留对团队有价值的文档
4. 完善的错误处理和日志输出
5. 定期审查和更新文档
